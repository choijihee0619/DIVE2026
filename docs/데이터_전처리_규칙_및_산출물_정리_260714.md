## 1. 한눈에 보기

데이터는 **3단계 레이어**로 관리한다. 각 단계의 역할이 다르므로, 파일을 옮길 때 목적에 맞는 폴더에 둔다.

```
raw  →  interim  →  processed
원본      표준화       학습·서빙용
```

| 레이어 | 역할 | 이번 단계 상태 |
|---|---|---|
| `raw/` | 원본 보존, API 응답 JSON 저장 | 발제사 8파일 이관 완료 |
| `interim/` | 컬럼명·타입·날짜·금액 표준화 | parquet 8개 생성 완료 |
| `processed/` | ML Feature, RAG 청크, 추론 결과 | RAG 완료, ML Feature는 미생성 |

---

## 2. 공통 전처리 규칙

모든 `interim/`·`processed/` 데이터에 적용하는 기본 규칙이다.

| 항목 | 규칙 | 예시 |
|---|---|---|
| 컬럼명 | `snake_case` | `auction_filed_date`, `subrogation_amount` |
| 날짜 | ISO `YYYY-MM-DD` | `2026-07-14` |
| 일시 | ISO + KST | `2026-07-14T13:30:00+09:00` |
| 금액 | **원 단위** `int` | `400000000` (4억) |
| 비율 | **0~1** `float` | `0.8073` (원본 80.73% → 0.8073) |
| 행 식별자 | 파일마다 `source_row_id` 부여 | UUID v4 |
| 원천 표시 | `source_system` 필수 | `hug`, `in`, `codef`, `mock` |
| 개인정보 | `processed/` 저장 전 마스킹·해시·UUID 치환 | 원문 저장 금지 |

### 파일명 규칙

```
{레이어}_{출처}_{YYYYMMDD}_{설명}.{확장자}
```

예: `interim_hug_20260714_임대보증_대위변제.parquet`, `raw_codef_20260714_case001.json`

- md5 해시로 이관 무결성 확인 완료

**API용 하위 폴더 갱신 (2026-07-14):** `scripts/collect_raw_data.py`로 9개 API(주소·실거래가·건축물대장·
사업자등록·DART·공시가격3종·CODEF)를 실호출 시도했다. 이 작업 환경의 아웃바운드 네트워크가
프록시 allowlist로 제한돼 전부 `network_blocked_by_sandbox_proxy`로 실패했고, 가이드 5장 Fallback
정책대로 Mock 응답을 `raw/{address,rtms,building,business_status,dart,official_price,codef}/`에
`raw_{api}_20260714_mock_fallback.json`로 저장했다(`source_system: mock`, 실패 사유 함께 기록).
**아직 실 API 응답이 아니다.** 아웃바운드 네트워크가 열린 환경에서 같은 스크립트를 재실행하면
성공한 API는 `*_live.json`으로 자동 교체된다. `raw/entities/`, `raw/onbid/`는 여전히 비어 있음
(entities는 발제사 제공 데이터에 법인명·사업자번호 식별자가 없어 배치 생성 불가, onbid는 키 미발급).

## 5. RAG 청크 규칙 및 산출물

| 항목 | 규칙 | 이번 적용 |
|---|---|---|
| 대상 | 아이엔 상담데이터, FAQ, 안내문 | 상담데이터만 처리 |
| Chunk 크기 | 500~900자 (한국어) | 적용 |
| Overlap | 80~150자 | 적용 |
| 제외 대상 | 주민번호, 전화번호, 계좌, 상세주소, 실명 | 자동 스캔 완료(2026-07-14 확장, 5종 패턴 전건 0건 검출) |
| Metadata | `doc_id`, `source`, `topic`, `consultation_stage`, `pii_removed` | 포함 |

**청크 JSONL 필드 예시** (`rag_chunks_260714_reviewed.jsonl` 기준)

```json
{
  "chunk_id": "rag_0001_0",
  "doc_id": "uuid",
  "source": "in_consulting",
  "topic": "경매·공매",
  "consultation_stage": "HUG이행청구",
  "text": "질문: ... 담당자의견: ...",
  "metadata": {
    "pii_removed": false,
    "pii_review_status": "자동 스캔 확장(전화번호/주민번호/이메일/계좌의심/상세주소) 완료, 사람 검수 필요 — 우선순위 low",
    "pii_scan_v2": {"hit_types": [], "hit_count": 0, "priority": "low", "human_reviewed": false}
  }
}
```

> ⚠️ `pii_removed: false` — 임베딩 전 **반드시 사람 검수** 필요. 2026-07-14에 전화번호·주민번호·이메일·
> 계좌의심패턴·상세주소(동/호/번지)까지 스캔 범위를 넓혔고 1,009건 전체에서 0건 검출됐으나, 자유서술형
> 문장의 실명·회사명·준식별자는 정규식으로 잡히지 않으므로 `pii_removed`를 자동으로 `true`로 바꾸지
> 않았다. 상세: `metadata/pii_review_260714.md`.

## 6. 아직 만들지 않은 것과 이유

### ML Feature 테이블 (`processed_risk_features_v1.parquet`)

RiskClassification 핵심 Feature는 아래 API·데이터 결합이 필요하다.

| Feature | 필요 데이터 |
|---|---|
| `jeonse_ratio` | 보증금 ÷ (실거래가 또는 공시가격) |
| `mortgage_ratio` | 근저당 채권최고액 ÷ 기준가격 |
| `rights_burden_ratio` | (보증금 + 선순위채권) ÷ 기준가격 |
| `has_seizure` | 등기부 압류·가압류 여부 |

→ 발제사 데이터에는 **등기부·실거래가가 없고**, 사고 **이후** 결과(대위변제·경매·배당)만 있다.  
→ API 키 발급 후 `raw/` 수집이 끝나면 `processed/ml/`에 Feature 테이블을 생성한다.

### RecoveryPrediction Label

`interim/dividend/`의 아래 컬럼은 **지금 바로 Label로 사용 가능**하다.

- `recovery_ratio` — 회수율 (0~1)
- `days_filing_to_dividend` — 신청일→배당 소요일

## 7. 알려진 제약사항

| 항목 | 내용 | 대응 |
|---|---|---|
| **파일 간 조인 불가** | 경매·배당에 `사고ID`·`사건번호` 없음 | 파일별 `source_row_id`만 사용, 조인 시도 금지 |
| **상수 컬럼** | `임대보증 사고현황`의 `incident_amount_ratio` 고유값 1개(100%) | Feature에서 제외 검토 |
| **주소 결측** | 0~24% (파일마다 다름) | 도로명주소 API로 법정동코드 정규화 예정 |
| **RAG PII** | 준식별자(계약일·지역·정황) 잔존 가능 | 임베딩 전 수동 검수 |
| **경매 상세 부재** | 감정가·최저가·낙찰가·법원 없음 | 필요 시 courtauction.go.kr 보완 검토 |
