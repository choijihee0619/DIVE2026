# HUG 백엔드 구현현황 (2026-07-23)

## 1. 범위와 결론

`HUG_UIUX_재설계_및_시연데이터전략_260723.md`에서 제안한 네 업무영역 중 백엔드를 구현했다. 이번 작업에서는 프론트엔드 코드를 변경하지 않았다.

- 통합 대시보드: 업무대장, 제공 합성 참조 포트폴리오, 공공 집계를 분리해 반환한다.
- 사고 전 계약관리: PU 사고위험 PoC, 계약별 예측 이력, D-90/60/30 증빙 bundle, 예방 케이스·조치·3자 알림을 제공한다.
- 사고접수·보증이행: 임차인 사고통지와 별도로 이행청구 원장을 생성하고, 서류·심사·명도·대위변제·채권등록·인계를 업무 액션으로 처리한다.
- 사고 후 채권관리: 등록채권 목록·병렬 상태축·회수 원장·회수전망 예측 이력·종결을 제공한다.
- 시연 데이터: 고정 기준일과 고정 ID를 사용하는 S1~S7 시나리오 및 manifest를 제공한다.

## 2. 구현 API

Base URL은 `/api/v1`이다. 아래 HUG 확장 API는 `hug_admin` 또는 `system_admin`만 호출할 수 있으며, 임차인 사고신고와 본인 이행청구 조회·서류제출은 기존 사용자 인증 규칙을 따른다.

### 2.1 통합 대시보드

| Method | Path | 용도 |
|---|---|---|
| GET | `/hug/dashboard/overview` | 업무대장·합성 참조·공공 집계 분리 KPI |
| GET | `/hug/dashboard/issuance-incident-trend` | 동일 상품·전국 발급/사고 연도 시계열 |
| GET | `/hug/dashboard/region-risk` | 시군구 사고율 공개 집계 |
| GET | `/hug/dashboard/issuance` | 보증발급 월별 공개 집계 |
| GET | `/hug/dashboard/victims` | 피해주택 공개 집계 |

사고 공개 원천은 연도 단위이므로 발급·사고 결합 API는 `requested_granularity=month`, `actual_granularity=year`와 폴백 사유를 반환한다. 월별 사고값을 임의 보간하지 않는다.

### 2.2 사고 전 계약관리

| Method | Path | 용도 |
|---|---|---|
| POST | `/ml/accident/predict` | 계약 원장을 입력으로 단건 PU PoC 추론·저장 |
| POST | `/hug/contracts/predictions/refresh` | 사고 전 계약 일괄 추론 |
| GET | `/hug/contracts` | 사고 전 계약 목록·필터·예방 우선순위 |
| GET | `/hug/contracts/{contract_id}` | 계약·예측·Rule·증빙·알림·타임라인 조합 상세 |
| GET | `/hug/contracts/{contract_id}/prediction` | 최신 계약 예측 |
| POST | `/hug/contracts/{contract_id}/prediction/refresh` | 단건 재예측 |
| GET | `/hug/contracts/{contract_id}/prevention` | 예방 케이스·bundle·조치 조회 |
| POST | `/hug/contracts/{contract_id}/preventive-actions` | 수동 예방조치 생성 |
| PATCH | `/preventive-actions/{action_id}` | 허용된 예방조치 상태전이 |
| POST | `/hug/contracts/prevention/sweep` | 고정 기준일을 지원하는 멱등 예방 점검 |

사고위험 출력은 `SUCCESS`, `NOT_SCORABLE`, `FAILED`를 구분해 계약별 이력으로 저장한다. 성공 결과에도 `calibration_status=AGGREGATE_PRIOR_ALIGNED_UNVALIDATED`와 PoC 근거를 포함하며 공식 사고를 자동 생성하지 않는다. 예방 우선순위는 사고위험 백분위 50%, 보증금 백분위 20%, 만기 긴급도 15%, 미해소 조치 심각도 15%의 제안식을 사용한다.

D-90/60/30은 각 3개 필수 항목을 별도로 추적한다. 한 문서만 제출해 전체 완료로 처리하지 않으며 `required/submitted/verified/overdue` 건수와 완료율을 저장한다. 알림에는 계약·예방 케이스·트리거·대상 역할·기한·전달/읽음/확인 메타데이터가 포함된다.

주요 상태전이는 `stage + version` 조건부 갱신으로 보호하며 경쟁 요청은 HTTP 409로 반환한다. 이행청구가 생성된 사고는 기존 4단계 사고상태 PATCH로 우회할 수 없다. 승인금액 초과 지급, 명도 전 지급, 대위변제 전 원금채권 등록, 원금채권·법무비용 증빙 전 소송대지급금 등록을 차단한다.

보증금 미반환과 경·공매 workflow는 명도 필요 여부가 다르므로 분기한다. SLA는 `DEMO_INTERNAL_V1` 시연 정책이며 HUG 공식 SLA가 아니다. 모든 업무 액션은 actor, role, request ID, 전후 단계와 사유를 감사 이벤트로 남긴다.

## 3. 데이터 출처와 시연 Seed

모든 신규 업무·예측 객체에는 다음 구조의 출처를 저장한다.

`data_mode`, `source_type`, `source_dataset`, `as_of_date`, `scenario_id`, `model_version`, `input_snapshot`, `is_demo`, `basis`

- 업무대장(`LIVE`/`DEMO`), 제공 합성 참조(`REFERENCE`), 공공 집계(`REFERENCE`)는 서로 합산하지 않는다.
- 제공 합성 CSV는 공통 사건 ID가 없으므로 주소·금액·날짜로 사건 단위 임의 조인하지 않는다.
- `POST /hug/demo/seed`와 `backend/scripts/seed_hug_workflow.py`는 template `hug-workflow-v1.1.0`, 고정 기준일 `2026-07-23`, 고정 ID, S1~S7 상태를 upsert한다.
- Seed는 계약·예측 요약만 만들지 않고 증빙요청/제출/검증, 이행청구 서류, 대위변제, 감사 이벤트, 회수 원장, 임차인·임대인·HUG 예방알림을 연결해 생성한다.
- `GET /hug/demo/manifest`는 template version, 시나리오 ID, 컬렉션별 건수, 문서 digest, 모델 artifact hash를 제공한다.
- Seed API는 `MOCK_MODE=true`에서만 허용한다. `--cached-predictions`로 오프라인 시연용 명시적 캐시 결과를 사용할 수 있다.
- 연결 조회용 MOCK 계정은 `workflow.tenant@example.com`, `workflow.landlord@example.com`, `workflow.hug@example.com`이며 공통 비밀번호는 `P@ssw0rd!`다.

## 4. 무결성·권한 보완

- 공개 회원가입으로 `hug_admin` 또는 `system_admin`을 생성할 수 없도록 제한했다.
- 사고통지 단건 조회는 `hug_admin`·`system_admin`과 신고 당사자만 허용한다. HUG 사고업무 권한 최소화를 위해 `advisor`의 사고 열람은 의도적으로 제외했다.
- 증빙 요청·제출·검증은 계약 당사자 또는 허용된 내부 역할만 접근할 수 있다.
- 예측 fingerprint, 알림 dedupe key, 증빙 bundle 항목, 이행청구/서류/지급/채권, 회수 이벤트·원장·예측에 고유 인덱스를 적용했다.
- 선택 입력인 dedupe/idempotency key는 compound sparse가 `null`을 고유값으로 오인하지 않도록 문자열 값에만 적용되는 partial unique index로 제한했다.
- 상태변경과 원장변경은 optimistic concurrency와 idempotency key를 사용한다.
- 회수 원장 `sequence`는 예약(PENDING) 단계에서 부여하지 않고, 채권 version CAS로 직렬화되는 `ledger_sequence` 카운터를 확정 시점에만 기록한다. 크래시로 남은 예약이 다른 멱등키 요청을 unique 충돌로 잠그지 못하며, receipt에 sequence를 함께 저장해 ambiguous write 복구 시 복원한다.
- 일괄 예측·예방 sweep의 LIVE/DEMO 모집단 필터와 명시 `contract_ids`는 `$and`로 결합한다. 명시 ID 지정이 모집단 분리 조건을 대체할 수 없다.
- HUG 전용 목록·업무 액션은 역할 검증을 적용한다.
- `data_mode`는 클라이언트가 지정하지 못하며 계약→이행청구→회수채권의 서버 원장에서 상속한다. LIVE와 DEMO KPI는 별도 집계한다.

## 6. 운영 전 남은 사항

1. 실제 HUG 정상 종료·사고 코호트로 PU PoC를 재학습·시간순 외부검증하고 실제 사고율로 확률 보정해야 한다. 현재 artifact 지원범위 밖 지역은 `NOT_SCORABLE`이다.
2. 사고위험 주요 요인은 건별 SHAP가 아니라 `ensemble_global_feature_importance`로 명시돼 있다.
3. 예방 sweep 서비스와 관리자 API는 구현했으나 실제 일일 cron/APScheduler 배포 설정은 별도다.
4. 이행서류는 파일 hash·object URI 메타데이터 원장이며 실제 object storage 업로드 경로 통합은 별도다.
5. 실제 운영은 예방·심사·승인·지급·채권등록·종결 권한과 maker-checker를 세분화해야 한다.
6. MongoDB 다중 컬렉션 transaction 대신 조건부 갱신과 보상 정리를 사용한다. 운영 환경에서는 outbox/reconciliation을 추가하는 것이 안전하다.
7. HUG 공식 SLA, 채권유형 원천코드, 원장 충당순서, 회수 우선순위 가중치는 발제사 확인 후 정책 버전을 갱신해야 한다.
